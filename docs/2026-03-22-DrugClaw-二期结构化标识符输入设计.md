# DrugClaw 二期结构化标识符输入设计

> 日期：2026-03-22  
> 当前阶段：方案设计  
> 核心目标：在 DrugClaw 入口稳定支持 `ChEMBL ID`、`PubChem CID`、`InChIKey` 三类结构化标识符输入，并统一归一化为 canonical drug name，再复用现有药名/别名标准化链路

---

## 1. 背景

一期已经解决了“标准药名 + 常见别名”的基础一致性问题，典型例子是：

- `imatinib`
- `Gleevec`

在固定 `mode + resource_filter` 下已经能稳定归一到同一个 canonical drug，并产出一致结果。

但从真实使用场景看，用户并不总是输入药名。接下来最常出现的第二类输入其实是结构化标识符：

- 论文或数据库里直接给 `CHEMBL941`
- 化学检索或表格里给 `PubChem CID 5291`
- 外部化学系统给 `InChIKey`

如果入口仍然只擅长处理自然语言和药名，那么这些输入会出现三个问题：

1. planner / retriever 继续猜输入含义，稳定性差
2. 不同 skill 对 ID 的识别能力不一致，导致结果分叉
3. 日志里无法清楚回答“系统到底把这个 ID 解析成了什么药”

所以二期的目标不是“让更多 skill 自己适配 ID”，而是把结构化输入解析前移到入口层。

---

## 2. 为什么二期先做这三类，而不是先做 SMILES

本期优先级定为：

1. `ChEMBL ID`
2. `PubChem CID`
3. `InChIKey`

不先做 `SMILES`，原因如下。

### 2.1 这三类输入更强结构化

- `CHEMBL941` 的模式很稳定
- `CID 5291` 的模式也很稳定
- `InChIKey` 有固定长度和分段格式

识别层可以依赖规则和正则完成，不需要 LLM 猜测。

### 2.2 这三类更容易映射到 canonical drug

- `ChEMBL` 可以直接返回 `pref_name`
- `PubChem` 可以返回 `Title`
- `InChIKey` 可以借助 `PubChem` 或上游数据库映射到具体化合物

这意味着入口层能较稳定地把“ID”还原成“药名”，再交给现有名称标准化层。

### 2.3 它们更适合做一致性验证

比如可以直接比较下面几条 query 最终是否一致：

- `What are the known drug targets of imatinib?`
- `What are the known drug targets of Gleevec?`
- `What are the known drug targets of CHEMBL941?`
- `What are the known drug targets of PubChem CID 5291?`
- `What are the known drug targets of KTUFNOKKBVMGRW-UHFFFAOYSA-N?`

### 2.4 SMILES 的失败面更大

`SMILES` 涉及：

- 盐型
- 立体化学
- 互变异构体
- 多组分表示
- 标准化和去盐

如果没有更完整的化学标准化栈，例如 RDKit 级别处理，直接上 `SMILES` 很容易把入口变成新的 bug 源。

结论：从“先把系统做稳”的角度，先做三类结构化标识符是更合适的二期。

---

## 3. 目标与范围

### 3.1 本期目标

- 支持在 query 入口识别 `ChEMBL ID`、`PubChem CID`、`InChIKey`
- 将结构化标识符解析成 canonical drug name
- 再复用现有 `drug_name_normalizer` 完成最终药名归一化
- 在 `query_logs` 中完整记录解析痕迹
- 在固定 `mode + resource_filter` 下，让“药名 / 别名 / 结构化标识符”查询尽量产出一致结果

### 3.2 In Scope

- 单药物单标识符场景
- 标识符嵌入自然语言问题中
- 入口级规则识别与外部解析
- 超时、失败、歧义的保守回退
- 自动化测试与人工回归测试

### 3.3 Out of Scope

- `SMILES`
- `InChI` 全串解析
- 多药物、多标识符混合比较问题
- 批量表格导入
- 化学结构级标准化
- 复杂盐型/前药/代谢物族谱折叠

## 4. 当前系统观察

结合现有代码，当前入口相关链路已经具备一期能力：

- [drugclaw/main_system.py](/data/boom/Agent/DrugClaw/drugclaw/main_system.py)
- [drugclaw/drug_name_normalizer.py](/data/boom/Agent/DrugClaw/drugclaw/drug_name_normalizer.py)
- [drugclaw/drug_alias_sources.py](/data/boom/Agent/DrugClaw/drugclaw/drug_alias_sources.py)
- [drugclaw/models.py](/data/boom/Agent/DrugClaw/drugclaw/models.py)
- [drugclaw/query_logger.py](/data/boom/Agent/DrugClaw/drugclaw/query_logger.py)

现状特点如下：

1. 已有统一入口药名标准化  
   `DrugClawSystem.query()` 会在进入 workflow 前做药名/别名归一化。

2. 已有 `normalized_query`、`resolved_entities`、`input_resolution` 三个状态出口  
   这让二期可以继续沿用现有状态面，而不用大改 workflow 接口。

3. 还没有结构化标识符解析层  
   当前系统没有一个专门负责“ID -> drug name”的模块。

4. ChEMBL 在检索层是现成资源，但这不等于入口层已经有 ID 解析能力  
   检索层的能力不应替代入口 canonicalization。

结论：二期应新增“结构化标识符解析层”，并把它放在 `drug_name_normalizer` 之前。

## 5. 设计原则

### 5.1 入口只做稳定解析，不做生成式猜测

对于结构化标识符，优先依赖：

- 规则识别
- 官方数据库返回

而不是让 LLM 猜这个 ID 可能是什么。

### 5.2 失败时可降级，不可崩溃

标识符解析失败时：

- 保留原始 query
- 标记 `input_resolution` 里的失败原因
- 让系统继续走原始 query 主链路

不能因为入口多了一层解析，就把整个系统变脆。

### 5.3 canonical drug name 仍然是系统主实体

后续 planner、retriever、logger、report 都应以 canonical drug name 为主。

结构化 ID 只是“用户输入形式”，不是系统内部的主运行实体。

### 5.4 保持状态面尽量小

二期优先复用一期已有字段：

- `normalized_query`
- `resolved_entities`
- `input_resolution`

不额外扩很多新的顶层状态字段，除非后续实现中发现确实有必要。

## 6. 总体架构

### 6.1 模块拆分

建议新增两个模块。

#### `drugclaw/drug_identifier_sources.py`

职责：

- 封装结构化标识符的外部解析能力
- 负责“查外部数据库并返回候选药名”
- 对不同来源做统一输出格式
- 负责超时、缓存、失败包装

建议包含三类 source adapter：

- `ChEMBLIdentifierSource`
- `PubChemIdentifierSource`
- `CompositeDrugIdentifierSource`

#### `drugclaw/structured_input_resolver.py`

职责：

- 识别 query 中是否出现结构化标识符
- 判断标识符类型
- 调用 `drug_identifier_sources` 做解析
- 输出统一的 identifier resolution payload
- 如解析成功，则将 query 中的标识符替换为 canonical drug name

### 6.2 入口链路

建议在 [drugclaw/main_system.py](/data/boom/Agent/DrugClaw/drugclaw/main_system.py) 中把当前入口链路改为：

`raw query`
-> `StructuredInputResolver`
-> `DrugNameNormalizer`
-> `workflow.invoke`

也就是：

1. 先判断用户是不是给了结构化标识符
2. 如果给了，就先解析成 canonical drug name
3. 再把得到的药名继续走一期的药名/别名标准化
4. 最终把合并后的结果写入 `AgentState`

### 6.3 为什么不把这层塞进 `drug_name_normalizer`

因为这是两个不同问题：

- `drug_name_normalizer` 解决“名字、别名、品牌名”
- `structured_input_resolver` 解决“结构化 ID 到名字”

混在一起以后，后面做 `SMILES` 或更多 ID 时会很难维护。

## 7. 输入识别与解析策略

### 7.1 ChEMBL ID

支持形式：

- `CHEMBL941`
- `chembl941`
- `ChEMBL:941`
- `drug CHEMBL941`

识别规则：

- 先做大小写归一化
- 抽取成标准形式 `CHEMBL941`

解析策略：

- 请求 ChEMBL molecule 接口
- 优先使用 `pref_name`
- 如果 `pref_name` 缺失，则回退到可用 synonym
- 解析结果统一 lower-case 后交给药名标准化层

设计动机：

- ChEMBL 对药物实体名的偏好字段稳定
- 本项目本身就广泛使用 ChEMBL 作为 DTI 来源

### 7.2 PubChem CID

支持形式：

- `CID 5291`
- `cid:5291`
- `PubChem CID 5291`
- `PubChem Compound 5291`

识别规则：

- 抽取纯数字 CID
- 标准化为字符串形式 `5291`

解析策略：

- 调用 PubChem PUG REST
- 优先取 `Title`
- 必要时回退到 synonyms
- 再交给药名标准化层

设计动机：

- PubChem CID 在论文补充材料和化学数据库中常见
- PUG REST 使用成本低，适合作为轻量入口解析源

### 7.3 InChIKey

支持形式：

- 纯 `InChIKey`
- `InChIKey=KTUFNOKKBVMGRW-UHFFFAOYSA-N`
- `InChIKey: KTUFNOKKBVMGRW-UHFFFAOYSA-N`

识别规则：

- 匹配标准 `14-10-1` 分段格式
- 大写规范化

解析策略：

- 优先通过 PubChem PUG REST 以 `InChIKey` 查询 CID/Title
- 如果得到药名，则继续进入药名标准化层
- 如果返回多个候选且归一后不一致，则标记 `ambiguous`

设计动机：

- InChIKey 是外部化学系统间非常常见的标准键
- 它本身适合作为入口标识符，但不适合作为系统内部最终主实体

## 8. 数据流设计

### 8.1 正常路径

以：

`What are the known drug targets of CHEMBL941?`

为例，理想链路如下：

1. `StructuredInputResolver` 识别到 `CHEMBL941`
2. 调用 `ChEMBLIdentifierSource`
3. 得到 `IMATINIB`
4. 将 query 重写成：
   `What are the known drug targets of imatinib?`
5. 再交给 `DrugNameNormalizer`
6. 得到最终：
   - `normalized_query = What are the known drug targets of imatinib?`
   - `resolved_entities = {"drug": ["imatinib"]}`
   - `input_resolution` 中同时包含 identifier trace 和 name trace

### 8.2 失败路径

例如：

`What are the known drug targets of CHEMBL_DOES_NOT_EXIST?`

处理原则：

- 不抛异常中断主链路
- `normalized_query` 维持原始 query
- `input_resolution.identifier_resolution.status = unresolved`
- 记录错误原因，如 `not_found`、`timeout`、`http_error`

### 8.3 歧义路径

如果一个结构化标识符返回多个名称候选，并且经过药名标准化后仍然不收敛到同一个 canonical drug：

- 标记 `identifier_resolution.status = ambiguous`
- 不强行改写 query
- 保留原始 query 继续主链路
- 在日志里完整记录候选项

保守优先于“硬猜一个结果”。

## 9. `input_resolution` 建议结构

二期不建议额外引入新的顶层 state 字段，而是扩展现有 `input_resolution`。

建议结构如下：

```json
{
  "original_query": "What are the known drug targets of CHEMBL941?",
  "normalized_query": "What are the known drug targets of imatinib?",
  "status": "resolved",
  "detected_drug_mentions": ["imatinib"],
  "canonical_drug_names": ["imatinib"],
  "alias_candidates": [],
  "resolution_trace": [
    {
      "matched_text": "imatinib",
      "matched_alias": "imatinib",
      "canonical_name": "imatinib"
    }
  ],
  "identifier_resolution": {
    "status": "resolved",
    "detected_identifiers": [
      {
        "type": "chembl_id",
        "raw_text": "CHEMBL941",
        "normalized_value": "CHEMBL941"
      }
    ],
    "resolved_records": [
      {
        "identifier_type": "chembl_id",
        "identifier_value": "CHEMBL941",
        "canonical_drug_name": "imatinib",
        "source": "chembl_rest"
      }
    ],
    "errors": []
  }
}
```

设计动机：

- 继续复用一期的状态出口
- 区分“标识符解析”和“药名解析”两层 trace
- 方便 query log、debug、线上回放

## 10. 外部数据源策略

### 10.1 总体原则

- 优先官方或稳定公开接口
- 只做轻量入口解析，不在这里做大规模检索
- 网络失败要快速超时
- 所有外部结果都要经过统一清洗

### 10.2 具体来源

- `ChEMBL ID -> drug name`
  - ChEMBL molecule REST
- `PubChem CID -> drug name`
  - PubChem PUG REST
- `InChIKey -> drug name`
  - PubChem PUG REST

### 10.3 缓存策略

建议在 resolver source 层引入轻量内存缓存：

- key: `(identifier_type, normalized_value)`
- value: 解析结果

缓存动机：

- 同一 query 在同一进程中多次查询时避免重复请求
- 后续 Web 服务化时能显著降低重复解析开销

本期不需要先做持久化缓存，内存缓存足够。

5. 解析失败或歧义原因

这对线上排查非常重要，因为以后 Web 服务化后，用户会直接拿各种 ID 来问。

- `CHEMBL941`


## 15. 风险与控制

### 15.1 第三方接口抖动

风险：

- ChEMBL 或 PubChem 临时慢、超时、限流

控制：

- 短超时
- 本地缓存
- 失败降级而不是中断

### 15.2 单个标识符返回多个候选

风险：

- 入口层误选错误药物

控制：

- 不强猜
- 标记 `ambiguous`
- 保留原始 query

### 15.3 名称源和结构化源不一致

风险：

- identifier 解析得到的名称与 alias 标准化结果不一致

控制：

- 统一再过一次 `drug_name_normalizer`
- 以最终 canonical drug name 作为唯一运行实体

---

## 16. 分阶段落地建议

建议把二期实现拆成四步：

1. 先做离线可测的模式识别和 source abstraction  
   先保证接口形状和失败回退逻辑正确。

2. 再接真实 ChEMBL / PubChem 解析  
   先把 `CHEMBL ID` 和 `CID` 做稳。

3. 再补 `InChIKey`  
   因为它依赖外部解析，但逻辑可以复用前两步的 source 层。

4. 最后做文档、人工回归和线上开关  
   为后续 Web 服务化打底。

---

## 17. 本期验收定义

二期完成的定义不是“系统能识别几个 ID”，而是下面四条同时满足：

1. `ChEMBL ID`、`PubChem CID`、`InChIKey` 都能走统一入口解析
2. 成功解析后，系统内部统一落到 canonical drug name
3. 失败或歧义时，系统不 crash，且日志可解释
4. 与标准药名/别名查询相比，结果一致性显著提升

满足这四条，二期才算真正为后续 `SMILES` 和 Web 服务化打好了基础。
