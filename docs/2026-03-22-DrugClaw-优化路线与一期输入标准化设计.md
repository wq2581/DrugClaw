# DrugClaw 优化路线与一期输入标准化设计

> 日期：2026-03-22  
> 当前阶段：优化方案设计  
> 核心目标：让 DrugClaw 先把“药物名及常见别名输入”这条链路做稳，再逐步扩展到更全面的输入覆盖与 Web 服务化

---

## 1. 背景与目标

当前 DrugClaw 已经具备：

- 多 agent 编排能力
- 多类药物资源技能
- evidence / claim / final answer 的中间层
- CLI 入口与 query logging

但从“真实可用系统”的角度看，仍有三类明显短板：

1. 稳定性不足  
   不是所有 query 都能稳定命中正确资源和实体。

2. 输入覆盖不足  
   当前主链路偏自然语言 query 和标准药名；同一药物的别名输入容易导致检索漂移。

3. 服务化不足  
   还没有可直接对外提供问答服务的 Web/API 层，也没有 token 配额治理。

本次设计不试图在一个版本里同时解决全部问题，而是先做 **第 1 期：药物名及常见别名输入标准化**。

---

## 2. 总体优化路线

整个优化工程建议拆成 3 期，而不是一口气混做。

### 第 1 期：输入覆盖打稳

目标：

- 支持药物标准名和常见别名输入
- 同一药物的标准名/别名尽量归到同一个 canonical drug
- 在固定 mode + resource_filter 下结果高度一致
- 标准化失败不导致系统崩溃

本期范围：

- 做 `drug name + alias`，不做 `SMILES / InChI / CID / ChEMBL ID`
- 不做 Web 前端
- 不做 token 配额系统

### 第 2 期：稳定性与输入类型扩展

目标：

- 扩展到 `SMILES / InChI / InChIKey / PubChem CID / ChEMBL ID`
- 收紧 planner / retriever / responder 的错误面
- 完善异常分类、超时治理、fallback 策略和结果一致性评估

### 第 3 期：Web 服务化

目标：

- 部署一个长期运行的 DrugClaw 服务
- 提供对话式 Web 页面和 API
- 默认走本地服务端模型/API 配置
- 增加 token 限额、速率限制、并发控制、服务日志与基本运维能力

---

## 3. 为什么第 1 期先做输入标准化

先做输入标准化有三个原因。

### 3.1 它是“稳定性”的前置条件

很多表面上的回答不稳定，本质上不是后面的 graph 或 responder 出错，而是：

- 同一个药物被系统当成了两个不同实体
- 某些 skill 用 brand name 能查到，某些 skill 只能吃 generic name
- planner/retriever 在别名输入下走了和标准名不同的路径

如果入口不能先把药物识别清楚，后面再堆更多优化只会放大噪声。

### 3.2 它最容易形成可验证的闭环

对于 `imatinib` 和 `Gleevec` 这种标准名/别名对，验收标准很明确：

- canonical drug 是否一致
- query plan 是否一致
- resource routing 是否一致
- 主结果是否一致

这类问题容易写自动化测试，也容易做人工验收。

### 3.3 它天然是后续输入扩展的底座

现在先做 `drug name + alias`，未来扩到：

- `SMILES`
- `InChI`
- `InChIKey`
- `PubChem CID`
- `ChEMBL ID`

时可以直接复用同一个输入解析框架，而不是再在各个 agent 里各补一层 if-else。

---

## 4. 第 1 期范围定义

### In Scope

- 药物标准名输入，如 `imatinib`
- 常见别名 / 品牌名输入，如 `Gleevec`
- 大小写、空格、简单符号噪声处理
- 高置信 typo 修正，如 `imatanib -> imatinib`
- 输入标准化结果写入执行链路和 query 日志
- 自动化测试与人工回归测试

### Out of Scope

- `SMILES / InChI / CID / ChEMBL ID`
- 多实体药物别名歧义交互确认
- Web API / Web 对话页面
- token 配额和账号体系
- 全局稳定性重构

---

## 5. 当前系统观察

结合当前代码，输入相关逻辑主要分布在：

- `drugclaw/main_system.py`
- `drugclaw/agent_planner.py`
- `drugclaw/agent_retriever.py`
- `drugclaw/entity_resolver.py`

现状特点：

1. `query()` 当前直接把原始 query 放入 `AgentState.original_query`  
   没有统一的前置输入标准化层。

2. `PlannerAgent` 和 `RetrieverAgent` 仍会从原始 query 中推断药物实体  
   对别名输入不够稳。

3. `EntityResolver` 当前主要负责 retrieval-side expansion  
   更像“帮助技能命中”的模块，而不是“全系统输入 canonicalization”的模块。

4. 仓库里没有现成可用的源文件版 `input_normalizer.py / structure_resolver.py`  
   不应假设已有成熟入口标准化层。

结论：

第 1 期应新增一个真正的入口标准化模块，而不是继续把职责堆到 planner / retriever / entity_resolver 里。

---

## 6. 设计原则

第 1 期设计遵守以下原则。

### 6.1 标准化模块只能提升稳定性，不能成为新的故障源

也就是说：

- 成功时提升命中率和一致性
- 失败时允许降级，但不能让系统 crash

### 6.2 入口标准化和检索侧扩展分层

- 入口标准化：负责“系统认准用户说的是哪个药”
- 检索侧扩展：负责“把这个药尽量在更多 skill 里查出来”

这两层不应混在一个模块里。

### 6.3 canonical name 是主实体，alias 是辅助信息

后续链路里：

- canonical drug name 是主输入
- alias 只作为补充召回或 trace 信息

### 6.4 原始 query 永远保留

系统可以在内部把 `Gleevec` 标准化成 `imatinib`，但不能丢掉用户原始输入。

---

## 7. 架构设计

### 7.1 新增模块

建议新增以下模块。

#### `drugclaw/drug_name_normalizer.py`

职责：

- 从 query 中识别药物 mention
- 对标准名、别名、常见拼写变体做解析
- 返回 canonical drug names、候选别名、置信度和 trace

#### `drugclaw/drug_alias_sources.py`

职责：

- 封装药物别名来源
- 第 1 版先提供：
  - 内置 alias seed
  - 可选 runtime source 接口

#### `drugclaw/models.py`

增加输入标准化结果结构，例如：

- `InputResolutionResult`
- `ResolvedDrugMention`

如果不想新开文件，也可以先直接放在 `models.py`，但要保持职责清楚。

### 7.2 入口接入点

入口接在：

- `DrugClawSystem.query()`

新链路：

`raw query -> drug name normalizer -> normalized query context -> workflow invoke`

### 7.3 状态传递

建议在 `AgentState` 里新增字段：

- `normalized_query: str`
- `resolved_entities: Dict[str, List[str]]`
- `input_resolution: Dict[str, Any]` 或结构化对象

含义：

- `original_query`：用户原始输入
- `normalized_query`：供 planner/retriever 优先消费的标准化 query
- `resolved_entities`：已确认的 canonical 实体
- `input_resolution`：完整 trace，用于日志和调试

## 8. 数据流设计

### 8.1 输入标准化流程

建议流程如下：

1. 接收用户原始 query
2. 做轻量文本清洗
3. 检测 query 中的药物 mention
4. 对每个 mention 进行标准化
5. 输出：
   - canonical drug names
   - alias candidates
   - resolution confidence
   - resolution source
   - resolution status

### 8.2 标准化优先级

建议按以下优先级解析：

1. 精确标准名命中
2. 精确别名命中
3. 轻量归一化后命中
4. 高置信 typo 修正

低置信模糊匹配不要强行替换。

### 8.3 后续消费优先级

Planner / Retriever 的优先级建议改为：

1. 优先用 `resolved_entities["drug"]`
2. 若不存在，再从 `normalized_query` 推断
3. 若仍不存在，退回原有 query 推断逻辑

这样可以最大程度减少 alias 引起的路由漂移。

---

## 9. 别名来源策略

### 9.1 第一层：内置 alias seed

第 1 版必须有一个小而稳的内置别名表，避免完全依赖外部资源。

建议首批包含：

- `imatinib -> Gleevec`
- `metformin -> Glucophage`
- `sildenafil -> Viagra`
- `atorvastatin -> Lipitor`

可以根据测试再扩几组，但不要一开始铺太大表。

### 9.2 第二层：可选 runtime 扩充源

在运行时可用的情况下，可从现有技能中补充 alias 信息：

- `RxNorm`
- `DrugBank`
- `FDA Orange Book`
- `openFDA Human Drug`
- `DailyMed`
- `MedlinePlus Drug Info`

但第 1 期不应把系统可靠性绑死在这些 runtime source 上。

### 9.3 来源优先级

建议优先级：

1. 内置 alias seed
2. 高质量 ontology / terminology source
3. knowledgebase / labeling source

不同来源若冲突，第 1 期先选择：

- 保守处理
- 记录歧义
- 不强改

---

## 10. Planner / Retriever / EntityResolver 的职责调整

### 10.1 PlannerAgent

调整后职责：

- 如果已有高置信 canonical drug，优先消费
- 只在没有 resolved entity 时，才从原始 query 再推断

这样 planner 从“猜测实体”转为“优先使用已标准化实体”。

### 10.2 RetrieverAgent

调整后职责：

- canonical drug name 作为主实体
- alias 仅作为补充候选
- 资源选择逻辑继续保留，但减少别名导致的 skill 漂移

### 10.3 EntityResolver

第 1 期不重写它，只重新定义它的职责边界：

- 它仍负责 retrieval-side expansion
- 不再承担 system-wide canonicalization 主职责

也就是说：

- `DrugNameNormalizer` 决定“这个药是谁”
- `EntityResolver` 帮忙“这个药还能用哪些高置信变体去查”

---

## 11. 错误处理与降级策略

输入标准化结果建议分三类：

### `resolved`

- 找到高置信 canonical drug
- 后续链路优先用 canonical name

### `ambiguous`

- 找到多个候选，但无法安全决断
- 记录候选到 trace
- 保留原 query 继续跑

### `unresolved`

- 没找到可靠映射
- 不报错
- 继续使用原 query

### 模块内部异常处理

标准化模块出现任何异常时：

- 吞掉异常
- 记录 `resolution_status=error`
- 降级为使用原 query

不允许输入标准化模块成为新的 crash source。

## 12. 可观察性与日志

建议把以下字段写入：

- `original_query`
- `normalized_query`
- `detected_drug_mentions`
- `canonical_drug_names`
- `alias_candidates`
- `resolution_status`
- `resolution_confidence`
- `resolution_source`
- `resolution_trace`

写入位置：

- `query()` 返回值
- `query_logs/<query_id>/metadata.json`

这样后续可以直接比较：

- `imatinib`
- `Gleevec`

是否在入口被解析成同一个 canonical drug。

## 13. 测试设计

### 13.1 单元测试

建议新增：

- `tests/input/test_drug_name_normalizer.py`

覆盖点：

- 标准名命中
- 别名命中
- 高置信 typo 修正
- 未知词不崩
- 大小写和空格噪声处理

### 13.2 组件测试

建议新增：

- `tests/system/test_query_input_normalization.py`

覆盖点：

- `query()` 能把标准化结果注入 `AgentState`
- planner/retriever 优先消费 canonical drug
- 标准化失败时系统仍可运行

### 13.3 人工回归测试

建议首批药物对：

- `imatinib` ↔ `Gleevec`
- `metformin` ↔ `Glucophage`
- `sildenafil` ↔ `Viagra`
- `atorvastatin` ↔ `Lipitor`

建议固定：

- `--mode simple`
- `--resource-filter`

先看：

- canonical drug 是否一致
- query plan 是否一致
- skills 是否一致
- 主结果是否一致

### 13.4 验收标准

第 1 期完成标准：

- canonical drug 一致
- query plan drug entity 一致
- selected skills 一致或高度一致
- 主 claim 高度一致
- 系统不因标准化失败崩溃

## 15. 与后续两期的衔接

第 1 期完成后，后续扩展会更自然。

### 对第 2 期

可以把 `DrugNameNormalizer` 演化成更通用的 `InputResolver`，扩展到：

- `SMILES`
- `InChI`
- `InChIKey`
- `PubChem CID`
- `ChEMBL ID`

### 对第 3 期

Web/API 层只需要调用统一的 `query()` 入口，无需自己处理别名问题。

这样输入标准化逻辑可以完全在服务端复用。

---

## 16. 风险与注意事项

### 风险 1：别名表过小导致覆盖不足

这是预期内问题。第 1 期目标不是全覆盖，而是先形成稳定框架。

### 风险 2：模糊匹配过激导致误归一化

第 1 期必须偏保守，高置信才改写，宁可漏改，不要错改。

### 风险 3：不同 skill 对 canonical name 的支持程度不同

这正是需要保留 alias 作为补充召回的原因。

### 风险 4：日志和状态过多导致链路复杂

第 1 期只加最必要字段，不做重型 tracing framework。

---

## 17. 最终结论

第 1 期不应直接去做所有输入格式支持，也不应先做 Web 服务化。

最合理的落点是：

- 先建立入口输入标准化层
- 先把药物标准名和常见别名打稳
- 让标准名与别名尽量归到同一个 canonical drug
- 通过自动化测试和人工回归验证结果一致性

只有这层打稳，后续再扩 `SMILES` 和 Web 服务化才不会持续叠 bug。
