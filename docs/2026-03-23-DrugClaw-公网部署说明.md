# DrugClaw 公网部署说明

> 日期：2026-03-23  
> 目标：把当前 DrugClaw Web/API 服务稳定地开放到公网，同时保证主查询链路不被部署方式拖崩。

---

## 1. 推荐拓扑

公网部署不要直接把 `drugclaw serve` 裸暴露到外网。推荐拓扑是：

```text
Browser
  -> https://your.domain.com
  -> nginx : 80/443
  -> 127.0.0.1:8000
  -> python -m drugclaw serve
  -> upstream LLM / resource APIs
```

这个拓扑的核心目的只有一个：

- `DrugClaw` 继续只监听本机地址
- `nginx` 负责公网入口、HTTPS 和反向代理
- 出问题时，服务层和部署层可以分开排查

如果你当前就是直接用一台本地 4090 机器对外提供 `drugclaw.site`，可以直接参考更具体的实操文档：

- `docs/2026-03-25-DrugClaw-4090直连公网部署方案.md`

---

## 2. 为什么不要直接暴露 8000

直接把 `8000` 端口暴露到公网虽然能用，但不适合作为正式部署方式。

主要风险：

- 没有 HTTPS
- 没有统一反向代理层
- 出现异常请求时缺少缓冲带
- 更容易把服务进程直接暴露给外部流量

对当前 DrugClaw 来说，稳定性优先，所以默认策略应当是：

- `drugclaw serve --host 127.0.0.1 --port 8000`
- 公网只开放 `80/443`

---

## 3. 上线前建议配置

第一版公网部署建议保持保守：

- `server_max_concurrency = 1`
- `server_query_timeout_seconds = 120`
- `server_default_mode = "simple"`
- `server_enable_graph_mode = false`

理由：

- 先保证服务不把上游模型额度和本地资源压崩
- 先保证浏览器可用、结果稳定、故障容易排查
- `graph` 模式更重，先不要作为默认公网入口

---

## 4. systemd 守护

仓库内已提供模板：

- `deploy/systemd/drugclaw.service`

使用前请至少确认下面三项：

- `User`
- `WorkingDirectory`
- `ExecStart`

如果你的部署路径不是 `/data/boom/Agent/DrugClaw`，要先改成真实路径。

常见启用流程：

```bash
sudo cp deploy/systemd/drugclaw.service /etc/systemd/system/drugclaw.service
sudo systemctl daemon-reload
sudo systemctl enable drugclaw
sudo systemctl start drugclaw
sudo systemctl status drugclaw
```

查看日志：

```bash
sudo journalctl -u drugclaw -f
```

---

## 5. nginx 反向代理

仓库内已提供模板：

- `deploy/nginx/drugclaw.conf`

使用前至少替换：

- `server_name your.domain.com`

常见启用流程：

```bash
sudo cp deploy/nginx/drugclaw.conf /etc/nginx/sites-available/drugclaw
sudo ln -s /etc/nginx/sites-available/drugclaw /etc/nginx/sites-enabled/drugclaw
sudo nginx -t
sudo systemctl reload nginx
```

---

## 6. HTTPS

推荐用 Let's Encrypt：

```bash
sudo apt-get install -y certbot python3-certbot-nginx
sudo certbot --nginx -d your.domain.com
```

完成后实际公网入口应当是：

```text
https://your.domain.com/
```

---

## 7. 防火墙建议

只开放：

- `80/tcp`
- `443/tcp`

不要开放：

- `8000/tcp`

例如：

```bash
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw deny 8000/tcp
```

---

## 8. 上线后验证

先在服务器本机验证：

```bash
curl http://127.0.0.1:8000/health
curl -X POST http://127.0.0.1:8000/api/query \
  -H 'Content-Type: application/json' \
  -d '{"query":"What are the known drug targets of imatinib?","resource_filter":["ChEMBL"],"save_md_report":true}'
```

再从公网域名验证：

```bash
curl https://your.domain.com/health
```

检查重点：

- `/health` 正常
- 首页能打开
- `/api/query` 能返回结果
- `query_logs/` 正常写入

---

## 9. 当前阶段不建议立刻做的事

下面这些功能以后可以做，但不建议和第一版公网部署绑在一起：

- 自定义 token 配额系统
- 多用户账号体系
- 分布式队列
- 多 worker 并发扩容
- 复杂前后端分离

原因很简单：这些都会显著增加复杂度，也更容易影响当前已经稳定的主查询链路。

---

## 10. 当前最重要的原则

DrugClaw 的主线任务仍然是：

- 接收问题
- 走现有 `DrugClawSystem.query()` 主链路
- 生成稳定、可追踪的查询结果

所以公网部署的设计原则不是“功能越多越好”，而是：

> 服务层尽量薄，部署层尽量稳，主查询链路尽量不动。
