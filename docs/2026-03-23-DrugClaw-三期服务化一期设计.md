# DrugClaw 三期服务化一期设计

> 日期：2026-03-23  
> 当前阶段：方案设计  
> 核心目标：在不破坏现有 CLI 主链路的前提下，为 DrugClaw 增加一个稳定、可长期运行的 Web/API 服务层，支持浏览器问答和基础运维保护

---

## 1. 背景与目标

当前 DrugClaw 已经具备：

- 可运行的 CLI 主入口
- 稳定的 `simple / graph / web_only` 查询模式
- 一期药名/别名归一化
- 二期结构化标识符输入归一化
- `query_logs` 持久化与 Markdown 报告

这意味着 DrugClaw 已经具备“核心查询能力”，但仍然缺少“服务能力”。

你当前的真实目标不是继续做一个本地命令行工具，而是要让它变成：

1. 服务器上长期运行的 DrugClaw 服务
2. 浏览器可直接使用的对话页面
3. 默认走服务端配置的模型/API key
4. 即使暂时不做自定义 token 限额，也不能把主任务搞崩

这个目标有一个关键约束：

> 第一版服务化，稳定性优先于功能丰富度。  
> 不能为了上 Web，把当前已经稳定的 CLI 主链路打乱。

---

## 2. 第一版服务化的核心原则

### 2.1 主链路零侵入

服务化第一版的基本原则是：

- CLI 继续保留，作为稳定基线
- `DrugClawSystem.query()` 仍然是唯一主查询入口
- Web/API 层只是外部包装，不重写主编排逻辑

也就是说，浏览器发起的问题，最后仍然由现有的：

- [drugclaw/main_system.py](/data/boom/Agent/DrugClaw/drugclaw/main_system.py)

负责执行主任务。

### 2.2 防崩优先，不先做复杂限额

用户已经明确：

- 现阶段可以不先实现自定义 token 限额
- 因为上游账号本身已有固定日限额

所以第一版不把重点放在“复杂计费系统”，而放在：

- 并发控制
- 超时控制
- 输入长度控制
- 错误隔离
- 服务健康检查

### 2.3 服务层独立，方便回滚

第一版服务化应该尽量新增文件，而不是在已有主链路上做大量侵入式改造。

设计动机：

- 万一 Web 层出现问题，可以快速停用而不影响 CLI
- 主查询逻辑和服务治理逻辑分层
- 后续换前端或扩 API 时不会污染主编排代码

---

## 3. 当前代码现状观察

当前仓库现状很明确：

- [pyproject.toml](/data/boom/Agent/DrugClaw/pyproject.toml) 只有 `langgraph`、`openai`、`requests`
- 没有现成 FastAPI / Flask / Gradio / Streamlit 服务层
- [drugclaw/cli.py](/data/boom/Agent/DrugClaw/drugclaw/cli.py) 是当前唯一正式入口
- [drugclaw/llm_client.py](/data/boom/Agent/DrugClaw/drugclaw/llm_client.py) 目前没有暴露 usage/token 统计

这意味着：

1. 第一版服务化不适合上来就做复杂前后端分离
2. “精确 token 配额”不是第一版的合适目标
3. 最合适的是先补一个轻量 HTTP 服务壳

---

## 4. 服务化方案选择

### 方案 A：FastAPI + 轻量前端

特点：

- 后端：FastAPI
- 前端：仓库内静态 HTML/CSS/JS
- 部署：`uvicorn` + `nginx` 反代

优点：

- 与当前 Python 项目最匹配
- 易于加健康检查、超时、并发保护
- API 和页面可以同时服务
- 后续升级到正式前后端分离成本可控

缺点：

- 前端体验不会像完整 React 应用那样丰富

### 方案 B：FastAPI + React/Vite 分离式前后端

优点：

- 更适合后续产品化
- 页面可扩展性最好

缺点：

- 初始工程量大
- 当前仓库无 Node/Web 前端基础设施
- 第一版更容易把重心从“稳定上线”拉偏

### 方案 C：Gradio/Streamlit 快速套壳

优点：

- 最快出 demo

缺点：

- 服务治理能力弱
- 路由、日志、限流、错误处理不够干净
- 不适合长期稳定运行

### 结论

第一版推荐：

> 方案 A：FastAPI + 轻量前端

原因很直接：

- 最符合当前仓库结构
- 改动最可控
- 最能满足“不要把主任务搞崩”

## 5. 一期服务化的功能边界

### In Scope

- 长期运行的 HTTP 服务
- 浏览器可用的最小聊天页面
- 调用现有 `DrugClawSystem.query()` 完成查询
- 基础健康检查
- 基础并发保护
- 基础超时保护
- 基础请求参数校验
- 历史 query 报告查看能力

### Out of Scope

- 登录账号体系
- 多租户
- 精确 token 计费
- Redis 分布式限流
- 完整流式 token streaming
- 多 worker 复杂任务队列
- 产品化聊天历史系统

---

## 6. 总体架构

### 6.1 后端分层

建议新增三层。

#### `server_app`

职责：

- 创建 FastAPI app
- 注册路由
- 提供静态页面
- 暴露健康检查

#### `service_layer`

职责：

- 封装对 `DrugClawSystem` 的调用
- 管理并发 semaphore
- 管理默认模式和默认资源过滤器
- 管理超时和错误包装

#### `api_models`

职责：

- 请求体/响应体 schema
- 参数校验
- 与 CLI 结果结构对齐

### 6.2 前端层

第一版前端只做一个轻量聊天页。

页面元素：

- Query 输入框
- 发送按钮
- 消息列表
- 处理中状态
- 折叠式调试信息面板
  - normalized query
  - query plan
  - claim assessments
  - report 链接

### 6.3 部署层

建议单机部署结构：

- `uvicorn` 跑 FastAPI
- `nginx` 反代
- `systemd` 兜底守护

原因：

- 足够支撑内部或小范围使用
- 出问题容易排查
- 与当前仓库复杂度匹配

---

## 7. 请求数据流

第一版服务化的数据流建议如下：

1. 浏览器提交 query 到 `/api/query`
2. API 层校验：
   - query 非空
   - 长度不超限
   - mode 合法
   - resource_filter 合法
3. Service 层获取并发许可
4. 调用现有 `DrugClawSystem.query()`
5. 得到与 CLI 接近的结构化结果
6. API 层包装为 JSON
7. 前端渲染结果

这个链路里最关键的一点是：

> 服务层不重新实现 query 逻辑，只是包裹和约束执行过程。

## 8. API 设计

### 8.1 `GET /health`

用途：

- 健康检查
- 部署验证
- 监控接入

返回建议包括：

- `status`
- `model`
- `default_mode`
- `service_uptime`
- `active_requests`

### 8.2 `GET /resources`

用途：

- 获取当前可用资源摘要
- 给前端展示默认/高级选项

返回建议包括：

- 当前资源总数
- ready/degraded/missing 数量
- 推荐默认资源组

### 8.3 `POST /api/query`

这是第一版核心接口。

请求体建议：

```json
{
  "query": "What are the known drug targets of imatinib?",
  "mode": "simple",
  "resource_filter": ["ChEMBL", "DGIdb", "Open Targets Platform"],
  "save_md_report": true,
  "include_debug": true
}
```

返回体建议：

```json
{
  "success": true,
  "query": "...",
  "normalized_query": "...",
  "resolved_entities": {"drug": ["imatinib"]},
  "input_resolution": {...},
  "answer": "...",
  "query_plan": {...},
  "claim_assessments": [...],
  "query_id": "...",
  "md_report_path": "...",
  "execution_stage": "...",
  "execution_trace": [...]
}
```

### 8.4 `GET /api/queries/{query_id}`

用途：

- 回看一次查询元数据
- 支撑前端历史查询详情

### 8.5 `GET /api/queries/{query_id}/report`

用途：

- 打开 Markdown 报告
- 前端“查看完整报告”按钮直连

## 9. 为什么第一版不先做流式输出

当前主链路还是一次性调用：

- `system.query(...)`

它没有天然暴露 token 级 streaming。

如果第一版强行做 SSE/WebSocket 流式输出，往往会带来两个问题：

1. 实现复杂度显著上升
2. 更容易把主任务链路搞乱

因此第一版建议：

- 不做 token streaming
- 前端采用同步等待或轮询式体验
- 先展示“处理中”状态

等服务稳定后，再考虑第二版：

- 阶段事件 streaming
- 最终答案 streaming

## 10. 防崩保护设计

### 10.1 并发保护

第一版建议非常保守：

- 最大并发数 = 1 或 2

原因：

- 当前主查询依赖 LLM 和多个外部资源
- 贸然放开并发，最容易导致上游 API 抖动和服务卡死

### 10.2 单请求超时

建议：

- HTTP 层设置请求超时
- Service 层设置 query 执行超时

超时后：

- 返回结构化错误
- 释放并发许可
- 记录错误日志

### 10.3 输入长度限制

建议：

- 限制 query 字符数
- 限制 `resource_filter` 数量

原因：

- 防止异常长输入拖垮系统
- 防止前端随意拼超大资源组合

### 10.4 模式白名单

第一版建议默认只开放：

- `simple`

可选地保留：

- `graph`

但不建议作为默认模式。

设计动机：

- `simple` 最稳定
- `graph` 更重，先作为高级选项

### 10.5 错误隔离

服务层必须捕获：

- 参数错误
- 上游超时
- 内部异常

并统一转换为结构化错误响应，而不是让后端直接抛栈打死请求。

---

## 11. 配额策略

### 11.1 第一版策略

用户已明确：

- 可以暂时不做自定义 token 限额
- 先依赖上游账号本身的日限额

因此第一版策略是：

- 不做精确 token 配额
- 只做“防崩式限制”
  - 并发
  - 超时
  - 输入长度

### 11.2 第二版预留

后续如果需要更细的使用治理，再补：

- LLM usage 采集
- 按总 tokens 统计
- 按 IP 或 session 限额
- Redis 持久化额度

这部分不应阻塞第一版上线。

---

## 12. 观测与日志

第一版建议保留现有 `query_logs`，并新增服务侧日志。

### 查询结果日志

继续沿用：

- `query_logs/<query_id>/...`

优势：

- 已有结构稳定
- 已能保存 Markdown 报告
- 与 CLI 和 API 共用同一套查询记录

### 服务运行日志

建议新增：

- 请求开始/结束
- 请求耗时
- 请求 IP
- mode / resource_filter
- 成功/失败状态

设计动机：

- 查询日志解决“答了什么”
- 服务日志解决“服务为什么慢/挂/失败”

## 13. 前端设计原则

第一版页面不要追求“像正式 SaaS 产品”，而要追求：

- 简单
- 稳
- 可解释

建议前端视觉和交互遵循：

- 单页聊天布局
- 明确的发送中状态
- 错误提示可读
- 可展开 debug 信息
- 直接链接到 report

第一版不做：

- 多会话管理
- 用户登录
- 复杂消息编辑
- 富文本复杂渲染

---

## 14. 实施顺序建议

建议严格按下面顺序做。

### 第一步：后端最小 API

先完成：

- `/health`
- `/api/query`

并用 curl/pytest 验证服务调用结果与 CLI 基本一致。

### 第二步：保护层

补上：

- semaphore 并发限制
- timeout
- 输入校验
- 错误包装

### 第三步：历史查询和报告接口

补上：

- `/api/queries/{query_id}`
- `/api/queries/{query_id}/report`

### 第四步：最小聊天页

前端直接消费 `/api/query`。

### 第五步：部署说明

补：

- 本地开发启动方式
- 服务器部署方式
- 反代和守护建议

## 15. 风险与控制

### 15.1 风险：服务层侵入主链路

控制：

- 新增服务模块
- 保持 CLI 入口不变
- 用 CLI 结果做回归基线

### 15.2 风险：多人同时使用导致卡死

控制：

- 低并发上限
- 明确超时
- 保守默认模式

### 15.3 风险：前端把复杂参数暴露过多

控制：

- 默认只展示最少参数
- 高级选项折叠
- 服务端做参数白名单校验

### 15.4 风险：上游 API 自身额度或可用性波动

控制：

- 第一版不承诺复杂限额
- 明确错误提示
- 保留服务运行日志

---

## 16. 一期完成定义

服务化一期完成，不是指“看起来像网页产品了”，而是同时满足：

1. 浏览器可以发起查询
2. 返回结果与 CLI 主链路基本一致
3. 服务异常不会把主任务链路拖死
4. 支持健康检查与基础排障
5. 部署和回滚都简单

如果这五条能满足，DrugClaw 就真正迈出了从“本地工具”到“可服务化系统”的第一步。
