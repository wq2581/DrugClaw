# DrugClaw 检索链路与输出优化规划

> 日期：2026-03-18  
> 适用范围：`python -m drugclaw run ...` 的默认 CLI 使用链路  
> 目标：在不重做架构的前提下，优先解决“慢、噪声多、输出不稳、答案不够像最终答案”的问题

## 1. 当前状态判断

结合当前代码和这次真实输出，可以先明确 3 个事实：

### 1.1 现在不是“优先本地调用”

当前系统不是简单地“优先本地资源”，而是更接近：

1. Planner / Retriever 先根据 query 选一组看起来相关、且当前 `ready` 的资源
2. 然后交给 Code Agent
3. Code Agent 对每个 skill 先做一次“生成查询计划”
4. 再做一次“生成可执行代码”
5. 再尝试执行
6. 执行失败后再 fallback 到 `skill.retrieve()`

从你这次的实际输出看：

- `ChEMBL`：CLI 型
- `Molecular Targets`：在线 API 型
- `DRUGMECHDB`：在线 API 型

这说明当前这次查询走的主要是 CLI / 在线资源，不是本地文件优先。

### 1.2 慢的主要原因不是单一资源慢，而是链路太长

你这次输出里最明显的时间开销点是：

- 每个 skill 都先进入 Code Agent
- 每个 skill 都先生成 code
- 然后 code 执行失败
- 再 fallback 到 `skill.retrieve()`

也就是说，当前简单查询其实走成了：

`选 skill -> 生成代码 -> 执行失败 -> fallback 检索 -> 汇总 -> 回答`

对于“imatinib 的已知靶点”这种非常标准的结构化查询，这条链路偏重了。

### 1.3 当前最终答案已经有证据，但输出仍然不够“产品化”

这次输出和之前相比已经进步了一步：

- 不再是 `0 structured evidence items`
- 已经有 `50 structured evidence items`
- 已经能产出 claim 和 evidence summary

但仍然有明显问题：

- 输出里保留了太多过程噪声
- claim 没有做足够的去重和清洗
- 混入了不适合作为“known drug targets”的项
- limitation 重复过多
- evidence summary 过于原始
- 最终答案仍然像“中间态调试输出”，不像面向用户的最终答案

---

## 2. 总体优化目标

后续修改建议按下面 4 个方向推进：

1. 让默认查询链路更短
2. 让检索选择更稳
3. 让最终答案更像“结论”，更少像“调试转储”
4. 让默认 CLI 更安静，把调试信息移到显式 debug 开关下

这里的核心原则是：

- 简单查询优先走最短路径
- Code Agent 只在真正需要时再启用
- 默认输出面向用户，而不是面向开发者
- debug 信息保留，但默认折叠

---

## 3. 推荐实施顺序

下面按优先级给出修改步骤。每一步都写明目的。

### 步骤 1：重构“简单结构化查询”的执行策略

**目的：先把最明显的延迟降下来。**

对于下面这类 query：

- known drug targets of X
- drug indications of X
- labeling / prescribing information of X
- direct DDI lookup

默认不应该先走“生成 code 再 fallback”。

建议改成：

1. Planner 识别 query 类型
2. 如果属于“直接结构化查询”
3. Retriever 直接调用 `skill.retrieve()`
4. 只有在需要复杂过滤、排序、字段拼装时才启用 Code Agent

建议增加一个明确的执行策略层，例如：

- `direct_retrieve`
- `constrained_code`
- `web_fallback`

其中：

- `target_lookup` 默认优先 `direct_retrieve`
- `labeling` 默认优先 `direct_retrieve`
- `mechanism synthesis` / `复杂跨字段筛选` 才进入 `constrained_code`

这一步做完后，像 `imatinib targets` 这种 query 应该先尝试直接检索，而不是每个 skill 都先生成代码。

### 步骤 2：把 skill 路由从“相关”收紧到“适合回答当前问题”

**目的：减少检索结果跑偏。**

你这次选出的 skill 是：

- `ChEMBL`
- `Molecular Targets`
- `DRUGMECHDB`

这里最大的问题不是有没有结果，而是“问题是 target lookup，但 skill 混入了 mechanism-oriented source”。

建议对 `QueryPlan.question_type` 增加更明确的 skill gating：

- `target_lookup`：
  只优先 DTI 直接证据资源
  例如 `ChEMBL`、`BindingDB`、`DGIdb`、`Open Targets Platform`
- `mechanism`：
  才引入 `DRUGMECHDB`
- `multi-hop reasoning`：
  才考虑 graph / mechanism KG

也就是说，后续不是只看“这个 skill 和 query 有点关系”，而要看“它是否是回答这个问题的一线证据源”。

### 步骤 3：在检索后增加“答案前清洗层”

**目的：让最终答案不再直接暴露原始 claim 噪声。**

你这次输出里最明显的问题包括：

- `IMATINIB has_ratio_activity Unchecked`
- `has_logp_activity No relevant target`
- `search_hit leukemia`
- `search_hit lymphoma`
- `K562`

这些内容可能是检索记录的一部分，但并不适合直接进入“known drug targets”的最终结论。

建议在 `ClaimAssessment` 之前或之中增加一个轻量清洗层：

1. 过滤明显非目标关系
2. 过滤占位值、空值、`Unchecked`、`No relevant target`
3. 区分：
   - 真正 target
   - cell line
   - disease
   - search hit
4. 对同义 target 做归并
   - 例如 `ABL` / `ABL1`
   - `KIT` / `Mast/stem cell growth factor receptor Kit`

这样最终 answer 才会从“原始三元组堆”变成“已知主要靶点列表”。

### 步骤 4：限制最终答案中的 claim 数量与类别

**目的：让答案聚焦，不再把 50 条证据全部展开成结论。**

对 `known drug targets` 这类问题，最终答案建议分层：

1. 主结论：只列 3 到 8 个最相关靶点
2. 次级证据：列少量可能的次要靶点或离体活性
3. 不确定项：单独放到“局限性/备注”

不要把所有 evidence item 都提升成 key claim。

建议增加规则：

- 主结论只允许 target/protein 类 claim 进入
- 同一 target 的重复 evidence 要折叠
- 单条弱证据不直接进入主结论
- 细胞系、疾病命中、搜索命中不能和 target claim 混排

### 步骤 5：重写默认 CLI 输出层

**目的：把默认体验从“研发调试台”改成“用户答案页”。**

当前默认输出仍然有很多研发态信息：

- `[Retriever Agent] ...`
- `[Code Agent] ...`
- fallback 提示
- 大量 limitation 重复
- Evidence Summary 太原始

建议把 CLI 拆成两层：

#### 默认层

只显示：

- query
- 简短执行摘要
- 最终答案
- 精简 evidence summary

#### 调试层

通过显式参数开启，例如：

- `--debug-routing`
- `--debug-code-agent`
- `--debug-evidence`
- `--debug-claims`

默认不打印这些 agent 级日志。

### 步骤 6：为 Code Agent 增加“跳过条件”

**目的：减少无意义的 code generation。**

当前 Code Agent 的问题不是不安全，而是触发范围太宽。

建议增加显式跳过条件：

- skill 只有标准 `retrieve()` 就够了
- query 是单实体直接查找
- 不需要复杂字段转换
- 不需要排序、聚合、交叉过滤

满足这些条件时：

- 不进入 code generation
- 直接 `skill.retrieve()`
- diagnostics 里标记为 `strategy=direct_retrieve`

只有下面情况才进入 `constrained_code`：

- 需要多字段组合过滤
- 需要复杂排序
- 需要结果裁剪和格式化
- 需要在 skill 返回结果上做安全的轻量变换

### 步骤 7：把 limitation / warning 做去重和分层

**目的：避免最终答案里出现几十行重复限制说明。**

你这次输出最大的问题之一是 limitation 几乎是“重复刷屏”。

建议改成分层聚合：

- 全局限制：
  - 主要依赖单源证据
  - 存在若干低上下文 snippet
- claim 级限制：
  - 仅对进入主结论的 claim 显示

不要为每条 evidence 都复制一遍“Claim relies on a single supporting evidence item.”

### 步骤 8：给“简单查询”单独加基准测试

**目的：防止之后又把默认路径改慢。**

建议为代表性简单查询加专门测试：

- `What are the known drug targets of imatinib?`
- `What prescribing information is available for metformin?`
- `How does warfarin interact with amiodarone?`

重点断言：

- 默认选中的 skill 是否合理
- 是否跳过不必要的 code generation
- 是否避免明显噪声 claim
- 最终答案是否保留结构化 evidence
- 默认输出是否不过度刷屏

## 4. 推荐的实际改造阶段

为了降低风险，建议按 3 个阶段落地。

### 第一阶段：性能和噪声控制

**目的：先改善体感。**

包含：

- 简单查询默认直连 `retrieve()`
- 默认关闭 agent 级调试日志
- 保留 debug flags
- 记录 `strategy=direct_retrieve` / `fallback_retrieve` / `constrained_code`

阶段完成标志：

- `imatinib targets` 不再每个 skill 都先生成 code
- 默认终端输出明显更短

当前状态（2026-03-18）：
- 已完成
- 当前默认链路已经改成简单查询优先 `direct_retrieve`
- agent 级调试日志已移到显式 `--debug-agents`

### 第二阶段：答案清洗与结论压缩

**目的：先让答案像答案。**

包含：

- claim 分类与过滤
- target 同义归并
- 限制项去重
- 只把高相关 target claim 放进主结论

阶段完成标志：

- 最终答案不再出现 `Unchecked`、`No relevant target`、`search_hit`
- limitation 不再刷屏

当前状态（2026-03-18）：
- 已完成
- `known drug targets` 主答案已改成清洗后的 `Known Targets`
- `Evidence Summary` 已优先使用过滤后的结构化证据
- 主答案现在优先展示更核心的靶点，并将其余 target-like activity 折叠到 `Evidence Summary`
- `Evidence Summary` 现已按最终 key claims 聚合，不再直接平铺原始 activity 命中

### 第三阶段：更稳的资源路由

**目的：进一步提高命中质量。**

包含：

- `question_type -> preferred resource family` 明确映射
- 机制类资源与 target 类资源分层
- 对简单 query 限制跨域 skill 混入

阶段完成标志：

- `target_lookup` 默认不再混入明显偏机制/搜索命中的资源

当前状态（2026-03-18）：
- 已完成
- `What are the known drug targets of imatinib?` 当前首轮路由为 `BindingDB`、`ChEMBL`、`DGIdb`
- `DRUGMECHDB` 和 `Molecular Targets` 已从这类直接靶点查询的默认前三中移除

## 5. 对当前问题的直接回答

### 问题 1：现在是优先本地调用吗？

不是。

当前更接近“优先当前 runtime 里看起来可用、且相关的资源”，并不是“本地文件优先”。你这次实际命中的 `ChEMBL`、`Molecular Targets`、`DRUGMECHDB` 主要是 CLI / 在线资源。

### 问题 2：为什么感觉联网搜索很慢？

更准确地说，慢的不只是“联网”，而是当前链路会：

1. 先给每个 skill 生成 query plan
2. 再生成 code
3. 再执行 code
4. 失败后再 fallback 到 `retrieve()`

这个开销对简单查询过大。

### 问题 3：是不是应该“先生成 code，再检索”？

对于复杂查询，可以。

但对于这类标准结构化查询，不建议默认这样做。更合理的是：

- 简单查询：先直接检索
- 复杂查询：再生成 code

也就是从“code-first”改成“retrieve-first, code-when-needed”。

---

## 6. 后续实施建议

建议后续按这个顺序推进：

1. 先改简单查询的执行策略，减少 Code Agent 触发
2. 再改默认输出，压掉调试噪声
3. 再做 claim 清洗和最终答案压缩
4. 最后再收紧 skill 路由规则

这个顺序的好处是：

- 先改善速度和体感
- 再改善答案质量
- 最后再提升路由精度

风险最小，也最符合当前仓库的稳定性优先方向。
